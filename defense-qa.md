# Ответы на вопросы для защиты работы

**Тема:** «Разработка приложения для выездных специалистов компании ООО "Водоучет"»

Документ соответствует фактической реализации в папке `Andrey_help` (React Native + Expo, Node.js + Express, lowdb).

**Навигация:** [README.md](README.md) · [structure.md](structure.md) · [concept.md](concept.md) · [instruction.md](instruction.md)

---

## Почему ты выбрал эту тему?

Тема выбрана, потому что у выездных специалистов компании «Водоучет» нет единого мобильного инструмента: заявки клиентов, назначение выездов, каталог услуг, отзывы и поддержка разрознены. Мобильное приложение объединяет эти процессы для **клиента**, **специалиста** и **администратора** в одном интерфейсе.

---

## В чём её актуальность?

- сотрудники работают вне офиса и нуждаются в мобильном доступе к заказам;
- клиенты ожидают онлайн-оформление заявок и отслеживание статуса;
- требуется разграничение прав (`admin` / `employee` / `client`);
- необходима защита персональных данных при регистрации (152-ФЗ — согласие и политика).

---

## Цель и задачи

**Цель:** разработать мобильное приложение для выездных специалистов ООО «Водоучет» с серверной частью и базой данных.

**Задачи:**

1. Реализовать роли `admin`, `employee`, `client`.
2. Обеспечить регистрацию клиента с валидацией и подтверждением email.
3. Реализовать каталог услуг с категориями и корзиной в заявке.
4. Добавить экраны: главная, каталог, контакты, профиль, отзывы, чат, админ; для клиента — заявки и заказы; для специалиста — выезды.
5. Хранить пароли в хэше (bcrypt).
6. Подготовить тестовые данные (6 пользователей, 9 позиций каталога, заявки, заказы).
7. Собрать APK, подготовить QR и план нагрузочного тестирования (JMeter).

---

## Теория

### Источники

- документация **React Native**, **Expo**, **Node.js**, **Express**;
- REST API, JWT-аутентификация;
- bcrypt для паролей;
- 152-ФЗ (общие принципы — согласие, политика, право на удаление).

### Подходы

- клиент–серверная архитектура;
- RBAC (роли на API и в UI);
- разделение слоёв: экраны → `api.js` → Express → lowdb;
- seed-данные для демонстрации на защите.

## Практика

### Данные

Seed при первом запуске (`server/src/db.js`):

- 6 пользователей (1 admin, 4 employee, 1 client);
- 9 позиций каталога, 3 категории;
- заявки (`incomingRequests`), заказы (`orders`), отзывы, чаты.

Рабочая БД: `server/data.json`.

### Результаты

1. Backend на порту 4000, REST API.
2. Mobile: Expo web, Expo Go, APK (EAS).
3. Проверка: `/api/health`, `/api/catalog`, сценарии по ролям.
4. JMeter — [docs/jmeter-plan.md](docs/jmeter-plan.md).

### Вывод

Приложение **работоспособно как учебный прототип** с полным циклом «заявка → заказ → выезд → отзыв». Для промышленной эксплуатации — облако, PostgreSQL, HTTPS (см. [ROADMAP.md](ROADMAP.md)).

## 1. О приложении в целом

### Какая задача решается?

Автоматизация работы выездных специалистов и клиентов: заявки, заказы, каталог, отчёты на объекте, админ-управление, чаты, отзывы.

### Для кого?

| Роль | Раздел | Функции |
|------|--------|---------|
| **client** | Мои заявки | Заявки, заказы, каталог, отзывы, чат, история |
| **employee** | Выезды | Назначенные выезды, отчёт, доп. работы |
| **admin** | Админ | Заявки, заказы, сводка, отчёты, каталог, чаты, клиенты |

### Реализованные функции

| Функция | Реализация |
|---------|------------|
| Роли | `users.role`, middleware `adminOnly` / `employeeOnly` / `clientOnly` |
| Регистрация | только client, валидация логина ≥4, пароля ≥8 + Aa + цифра |
| Пароли | bcrypt → `password_hash` |
| Email | `/api/auth/verify`, `/api/auth/forgot-password` |
| ПДн | чекбокс + `PrivacyScreen`, удаление аккаунта |
| Каталог | 9+ карточек, фильтр, «В заявку», CRUD в админке |
| Заявки/заказы | полный цикл согласования, таймлайн, чат по заказу |
| Выезды | `OrdersScreen`, отчёт, причины отказа |
| Отзывы | список, клиент по завершённому заказу, ответ админа |
| Чат | `SupportChatWidget`, чат по заказу, inbox админа |
| Сортировка списков | `ListSortChips`: новые, старые, активность, дата выезда |
| Подтверждения | «Да / Нет» при отмене заявки (`ConfirmHost` на web) |
| Адаптивность | сетка каталога, web/native контакты |

### Почему mobile, а не только сайт?

Специалист на объекте использует телефон. Клиенту удобнее мобильный сценарий «заявка в два клика». Web-версия есть для разработки и демо (`expo start --web`).

## 2. О технологиях

| Технология | Зачем |
|------------|-------|
| **React Native + Expo** | Кроссплатформа Android/iOS, одна кодовая база |
| **Node.js + Express** | REST API, единый язык с клиентом |
| **lowdb** | Файловая JSON-БД без отдельного СУБД-сервера |
| **JWT** | Сессия после входа |
| **bcrypt** | Хэш паролей |
| **EAS Build** | Сборка APK в облаке |

**Почему не Firebase?** Задача — показать **свой** backend, свои API, роли и нагрузочное тестирование.

**Почему не PostgreSQL в прототипе?** Проще перенос и демонстрация (один файл `data.json`). Для production — PostgreSQL ([ROADMAP.md](ROADMAP.md)).

## 3. Архитектура

```
[Mobile: React Native / Expo]
        ↓ HTTP/JSON (REST)
[Backend: Node.js + Express]
   ├── index.js      — auth, catalog, orders, client, admin
   ├── chatRoutes.js — чаты
   ├── roadmapRoutes.js — дашборд, жалобы, профиль, …
   └── audit.js      — журнал
        ↓
[lowdb → server/data.json]
[Статика → server/public/catalog/]
```

### Общение с сервером

`mobile/src/services/api.js` → `fetch` к `EXPO_PUBLIC_API_URL` (по умолчанию `http://localhost:4000/api`).

### Хранение данных

| Данные | Где |
|--------|-----|
| Пользователи, заявки, заказы, чаты | `server/data.json` |
| Картинки | `server/public/catalog/` |
| JWT после входа | память приложения (`AuthContext`) |

### Защита

- пароли: bcrypt;
- API: JWT + проверка роли;
- отмена/удаление: подтверждение пользователя;
- аудит и логи ПДн: `auditLog`, `pdAccessLog`.

### Без интернета

Каталог и API не загрузятся (офлайн-кэша нет). Приложение показывает сообщение об ошибке, не падает.

## 4. О разработке

### Трудности

1. Подключение телефона к локальному backend (IP, firewall).
2. EAS Build — очередь, настройка `EXPO_PUBLIC_API_URL`.
3. Web: `react-native-maps` → отдельный `ContactsScreen.web.js`.
4. Объединённый список заявок и заказов — разные id-счётчики → сортировка по `created_at`.
5. Race condition при выходе — исправлено `tokenRef` в `AuthContext`.

### Тестирование

- ручные сценарии по ролям;
- `/api/health`, `/api/catalog`;
- Expo web, Expo Go, APK;
- JMeter — [docs/jmeter-plan.md](docs/jmeter-plan.md).

### APK

```bash
cd mobile
eas build -p android --profile preview
```

Подробно: [intoapp.md](intoapp.md), [docs/release-and-qr.md](docs/release-and-qr.md).

## 5. Сеть и запуск

### Почему не работает без ПК?

Backend локальный. APK обращается к IP ПК в Wi‑Fi.

### Как исправить?

Облачный сервер + HTTPS + постоянный URL ([ROADMAP.md](ROADMAP.md)).

### Почему пересборка APK при смене IP?

`EXPO_PUBLIC_API_URL` вшивается при сборке. Альтернатива — экран настроек сервера в приложении.

### Подключение телефона

1. Одна Wi‑Fi с ПК.
2. Backend на порту 4000.
3. `EXPO_PUBLIC_API_URL=http://<IP>:4000/api`.

## 7. База данных

Коллекции в `data.json`:

| Коллекция | Назначение |
|-----------|------------|
| `users` | пользователи, роли |
| `categories`, `catalogItems` | каталог |
| `incomingRequests` | заявки |
| `orders` | заказы на выезд |
| `reviews` | отзывы |
| `chatMessages` | чаты |
| `complaints` | претензии |
| `auditLog`, `pdAccessLog` | аудит, ПДн |

Связи: `orders.client_user_id`, `orders.assigned_user_id` → `users.id`; `reviews.order_id` → `orders.id`; `catalogItems.category_id` → `categories.id`.

Миграций нет — seed при пустой БД, правки через `data.json` или админку.

## 10. Провокационные вопросы

**Украли телефон?** Без пароля доступ ограничен; JWT с ограниченным сроком; в production — блокировка, биометрия.

**Взлом сервера?** Учебный локальный стенд; пароли — bcrypt-хэши; в production — HTTPS, firewall, бэкапы.

**Почему не Kotlin/Swift?** React Native — одна база, быстрее для курсовой, нативный UI (не WebView).
